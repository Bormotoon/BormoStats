# Security Policy

## Supported Branch

Security fixes are expected to land on `main`.

## Reporting a Vulnerability

Do not open a public GitHub issue for security-sensitive findings.

Preferred process:

1. Use GitHub private vulnerability reporting if it is enabled for the repository.
2. If private reporting is not available, contact the maintainer out of band.
3. Include a clear impact summary, affected area, reproduction steps, and any suggested mitigation.

## What to Include

- Affected component or file path
- Preconditions and attack surface
- Reproduction details
- Observed impact
- Suggested remediation, if available

## Sensitive Data Handling

- Never include real seller tokens, admin keys, ClickHouse passwords, or Telegram credentials in a report.
- Redact marketplace account identifiers unless they are required to explain impact.

## Response Guidelines

- Reports should be acknowledged promptly.
- Public disclosure should wait until a fix or mitigation is available.
